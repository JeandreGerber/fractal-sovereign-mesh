# Contributing to the Fractal Sovereign Mesh Protocol

Thank you for being here. This project is only as good as the people who build it.

The FSM Protocol is an open-source framework — the philosophy, architecture, blueprints, governance models, legal templates, and eventually the app are all built collectively. The core maintainers hold the vision and protect the immutable principles. Everyone else shapes everything else.

---

## What We Need

The protocol has real unsolved problems that require domain experts. See [problems/OPEN_PROBLEMS.md](problems/OPEN_PROBLEMS.md) for the current list. If you have expertise in any of the following areas, your contribution is immediately valuable:

- **Engineering** — water recovery, energy systems, hydroponic design, construction
- **Legal** — cross-jurisdiction corporate structure, housing equity as securities, cooperative law
- **Medicine** — autonomous diagnostic systems, community health protocols
- **Agriculture** — permaculture, hydroponics, food forest design, soil biology
- **AI / Software** — governance systems, local-first architecture, readiness scoring algorithms
- **Organizational Psychology** — conflict resolution at scale, Dunbar-stack governance design
- **Architecture / Urban Planning** — modular construction, underground infrastructure, edible urbanism

## The Two Layers — What Is and Is Not Open to Change

### The Immutable Layer (Not open to modification proposals)

The contents of `/foundation/charter.md` — the nine immutable principles — are the foundation the entire system rests on. They are not open to modification PRs. If you believe a fundamental principle is wrong, open a Discussion thread. Changing these requires a process that does not exist yet and will require broad community consensus when it does.

### The Plugin Layer (Everything else — fully open)

Everything outside the immutable principles is open to proposals:
- Blueprint specifications
- Governance protocols
- Legal templates
- Training curriculum
- App features and architecture
- Node formation guides
- System specs (energy, water, food, waste)

## How to Contribute

### 1. Opening a Problem

If you identify a gap, flaw, or unsolved question in the protocol:

1. Check [problems/OPEN_PROBLEMS.md](problems/OPEN_PROBLEMS.md) — it may already be listed
2. Check existing [Issues](../../issues) — it may already be open
3. Open a new Issue with the label `problem`
4. Use this format:

```
**Problem:** One clear sentence describing what is unsolved.

**Why it matters:** What breaks in the system if this stays unsolved.

**Domain:** Which discipline this falls under (engineering / legal / medical / etc.)

**What a good solution looks like:** Any constraints or requirements the solution must meet.
```

### 2. Proposing a Solution

1. Fork the repository
2. Create a branch: `solution/your-problem-name`
3. Add your solution in the relevant directory
4. Open a Pull Request referencing the Issue it addresses
5. Describe your solution clearly — what it does, what assumptions it makes, what it does not solve

PRs that touch `/foundation/` will not be merged. Everything else is fair game.

### 3. Joining a Discussion

GitHub Discussions is where the big questions live — architecture decisions, philosophical debates, "what if we did it this way" proposals. Before writing significant code or documentation, post a Discussion thread to get input. This prevents duplicate work and keeps the protocol coherent.

Good Discussion topics:
- Proposing a new directory or structural change to the repo
- Questioning an architectural assumption in the manifesto
- Sharing a real-world implementation or experiment
- Proposing a new open problem

### 4. Good First Contributions

New here? Look for Issues labeled `good first issue`. These are well-defined, contained tasks with clear acceptance criteria. They are a good way to understand the protocol and get familiar with how contributions work before tackling harder problems.

## Pull Request Standards

- One PR per problem or feature — do not bundle unrelated changes
- Include a clear description of what changed and why
- Reference the Issue or Discussion the PR addresses
- If your PR touches a spec that affects other specs, flag the dependency
- PRs that modify immutable principles will be closed without merge

---

## Code of Conduct

This project is built on the same philosophical foundation as the towns it describes: interconnectedness, reciprocity, and sovereign responsibility. In practice, that means:

- Engage in good faith — assume the best interpretation of someone's proposal before critiquing it
- Disagree with ideas, not people
- No discrimination, harassment, or bad-faith engagement
- If a conflict arises, open a Discussion thread — do not resolve it in PR comments

Violations result in removal from the contributor community. The exile protocol applies here too.

---

## A Note on the Immutable Principles

Every system eventually faces someone who wants to remove a core constraint because it is inconvenient. The nine immutable principles exist because history shows that every community that compromised its foundational values eventually became what it was built against.

If you find yourself wanting to propose removing "One Resident, One Vote" to make governance more efficient, or removing the "Anti-Empire Clause" because a larger ownership structure seems more practical — that is exactly the kind of proposal the immutable layer exists to prevent. The principles are not obstacles. They are the reason the system works.
